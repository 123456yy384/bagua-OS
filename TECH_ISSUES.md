# BaGua OS — Open Technical Challenges

> Discussion checklist. Each issue identifies the modules involved and the scope of impact.

---

## 1. Polarity Vector Hardware Encoding

**Modules:** scheduler / polarity / hal::device

AI-layer polarity vectors are generated by a learnable matrix (Linear + Tanh). The OS layer has no backpropagation — polarity must be **hardcoded or heuristically derived** from hardware features.

**Decision needed:**
- What does each of the 32 vector dimensions represent physically? (compute density, power curve slope, cache locality, IO bandwidth…)
- How to expand a device capability_vector into 32 dimensions?
- Is runtime online learning needed (e.g., statistical clustering of process execution history)?

**Impact:** Impedance matrix accuracy directly determines scheduling efficiency. Wrong polarity encoding → broken impedance matrix → devices that should collaborate get isolated, ones that should be isolated get bound together.

---

## 2. Causal Mask Semantics at OS Level

**Modules:** scheduler

The AI-layer causal mask means "position t can only receive information from before t." What is the OS equivalent?

**Possible interpretations:**
1. **Event causality:** an interrupt handler's result cannot retroactively modify the syscall that triggered the interrupt
2. **Resource directionality:** CPU → GPU data transfer is a one-way pipeline
3. **Security boundary:** low-security subsystems cannot propagate information back to high-security ones

**Decision needed:** Which interpretation(s) apply, or all three?

## 3. Jiuzhou Address vs. MMU

**Modules:** memory

The Jiuzhou three-level address structure `[slot_id][object_id][offset]` has two implementation paths:

| Approach | Description | Problem |
|----------|-------------|---------|
| Pure Jiuzhou | Bypass MMU, custom page tables | Incompatible with existing hardware |
| MMU-compatible | Jiuzhou address → standard page table → physical address | Extra translation step |

**Decision needed:**
- How to implement on x86_64? (must use MMU + page tables)
- Can RISC-V bare-metal customize TLB logic directly?
- How does the Linux compatibility layer translate between the two address types?

---

## 4. Left-In Right-Out: Process State Zeroing

**Modules:** memory

"Accumulate within a sequence, zero out between sequences" is far more complex at the OS level than at the AI level:

- `fork()` should NOT copy page tables → how does the child get its context?
- If IPC must use message passing → what is the overhead of message passing?
- What do the 32 memory slots represent at the process level? (register snapshots? TLB entries? security enclaves?)

**Core contradiction:** Aggressive isolation provides security benefits but may make the traditional `fork + exec` model unworkable.

---

## 5. Elimination Audit "Output Quality" Definition

**Modules:** audit

The AI layer's elimination audit depends on "output quality" (logical_density × causal_stability). The OS layer needs to define what a process's "output" is:

- IO bytes? → idle processes also produce zero IO, would be falsely killed
- Syscall frequency? → infinite loop calling `getpid()` also has high frequency
- Page writeback count? → compute-intensive processes have low writeback but are useful
- A composite metric?

**Decision needed:** The quality-scoring formula, and whether processes can register custom "quality reports" (userspace-defined audit metrics).

---

## 6. Logic Pressure vs. Elimination Threshold Coupling

**Modules:** self_comp / elimination

When `logic_pressure` rises (survival rate drops) → how should the elimination threshold change?

| Intuitive Approach | Problem |
|--------------------|---------|
| Raise threshold | Harder to survive, more processes eliminated → pressure may keep rising (vicious cycle) |
| Lower threshold | Easier to survive → many low-efficiency processes persist → system slows down |

**Decision needed:**
- When pressure rises: stricter elimination or looser tolerance?
- The AI layer's feedback loop uses `detach()` to block gradients — does the OS layer need an equivalent?
- Cascade protection: one subsystem's survival rate hitting zero shouldn't mean global collapse.

---

## 7. Carbon-Based Chip Scheduling Layer Interface

**Modules:** hal::device (Slot 1)

Carbon-based chips don't replace transistors — BaGua architecture runs on carbon-based chips as a scheduling layer. But how exactly does the interface work?

- Carbon chip neuromorphic arrays have their own programming model (spike timing? SNN?)
- How does the BaGua kernel scheduler run on carbon chips — compiled to carbon instruction set, or only a monitor daemon on the carbon chip?
- Does the carbon chip's physical slow-discharge characteristic naturally fit the computational buffer? If so, how to quantify?

---

## 8. Task Perception: "First Syscall" Semantics

**Modules:** task_gate

The AI layer determines task type from the first token. The OS equivalent is judging from a process's **first syscall**, but:

- Many processes' first syscall is `mmap` (dynamic linker) — almost useless for task classification
- If we wait for the first N syscalls → what value of N? Is the delay acceptable?
- A process may change behavior patterns mid-execution (compile first, analyze later) → does `causal_strength` need mid-course adjustment?

**Decision needed:**
- Sampling window size
- Whether the kernel is allowed to reassess mid-process and switch scheduling policies

## 9. Multiboot2 Magic Byte Order

**Modules:** kernel/src/main.rs

The Multiboot2 magic constant is `0xE85250D6`. The original code had the byte order reversed to `0xE86E36D6`, causing QEMU/Limine to reject the kernel.

**Fixed:** 2026-05-25, `main.rs:30`.

## 10. QEMU -kernel 64-bit Boot Compatibility

**Modules:** Build system / Deployment

QEMU's `-kernel` option has fragmented 64-bit kernel boot support across versions:

| QEMU | MB1 (32-bit) | MB2 (64-bit) | PVH |
|------|-------------|-------------|-----|
| 6.x | ✅ | ❌ | ❌ |
| 7.x | ✅ | ❌ | ❌ |
| 8.x+ | ❌ | ❌ | ❌ (specific format required) |

**Conclusion:** Bare-metal verification should not depend on QEMU `-kernel`. Use Limine/Grub2 bootable images, or real hardware UEFI boot.

---

## 11. 32→64 Long Mode Transition

**Modules:** kernel/src/boot64.S + kernel/build.rs

Multiboot2 boots in 32-bit protected mode. A 64-bit kernel must switch to long mode itself. The trampoline is implemented in boot64.S, assembled via GNU as (WSL), and linked by build.rs.

**Status:** Trampoline compiles and links successfully. Awaiting Limine boot verification.
